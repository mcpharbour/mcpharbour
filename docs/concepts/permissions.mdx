## Default Deny

MCP Harbour uses a **whitelist-based** permission model. An identity with no policy gets zero tools — the harbour creates an empty policy that denies everything.

## How Policy Evaluation Works

When an agent calls a tool, the harbour checks three things in order:

<Steps>
  <Step title="Tool lookup">
    The harbour resolves which server owns the requested tool. If the tool doesn't exist on any server the identity has access to → `AUTHORIZATION_DENIED`.
  </Step>
  <Step title="Tool check">
    Does any tool permission in the policy match the requested tool name? If not → `AUTHORIZATION_DENIED`.
  </Step>
  <Step title="Argument check">
    If argument policies are defined and arguments are provided, do all policies pass? If any fail → `AUTHORIZATION_DENIED`.
  </Step>
</Steps>

If all checks pass, the request is forwarded to the MCP server.

## Tool Filtering

`list_tools` only returns tools that match at least one entry in the policy. Agents never see tools they aren't allowed to use.

## Argument Policies

Argument policies restrict the values of specific tool arguments. Two match types are supported:

| Type | Syntax | Example |
|------|--------|---------|
| **Glob** (default) | `arg=pattern` | `path=/home/user/**` |
| **Regex** | `arg=re:pattern` | `sql=re:^SELECT\s.*` |

A glob pattern without wildcards is an exact match — `mode=readonly` matches only `"readonly"`.

### CLI Examples

```bash
# Glob: allow read_text_file only under /home/user/projects
harbour permit allow agent filesystem --tool "read_text_file" --args "path=/home/user/projects/**"

# Regex: allow only SELECT queries
harbour permit allow agent database --tool "query" --args "sql=re:^SELECT\s.*"

# Multiple argument policies on one tool
harbour permit allow agent database --tool "query" --args "sql=re:^SELECT\s.*" "db=production"
```

## Identity

Identities are created with `harbour identity create <name>`. Each gets:

- A **name** (stored in `config.json`)
- A **key prefix** for display (stored in `config.json`)
- A **bcrypt-hashed API key** (stored in system keyring — never in config files)

The agent authenticates by sending the API key as `Authorization: Bearer <key>` to the daemon's Streamable HTTP MCP endpoint. The daemon derives the identity by checking the token against all stored hashes.

<Warning>
  The agent does not declare its identity. The daemon derives it from the token. The agent cannot influence how it is identified.
</Warning>
